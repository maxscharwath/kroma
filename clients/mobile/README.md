# @kroma/mobile

KROMA mobile client for iPhone, iPad and Android, built with Expo (React
Native, New Architecture) on top of the shared `@kroma/core` API client, wire
types and i18n catalogs.

## Run it

```sh
bun install                    # repo root
cd clients/mobile
bun run start                  # Metro dev server (scan the QR with a dev build)
bun run ios                    # prebuild + compile + launch on the iOS simulator
bun run android                # prebuild + compile + launch on an Android device/emulator
```

Native projects (`ios/`, `android/`) are generated by `expo prebuild` and
gitignored; edit `app.json` + config plugins, never the generated projects.
Node must be installed: the Expo CLI's template extraction is broken under the
bun runtime (NUL-padded files), and `bunx` prefers Node when present. Metro
bundling itself works fine either way.

The workspace pins `@babel/core@^7` locally because the repo root hoists
Babel 8 (web client) while the React Native toolchain requires 7.

## Playback engine

`src/player/engine/` ports the TV clients' base-engine model (see
`packages/tv/src/features/playback/player/baseEngine.ts`) onto expo-video
(AVPlayer on iOS, ExoPlayer on Android). `types.ts` holds the vocabulary and the
mutable core, `index.ts` owns loading + the player events, `controls.ts` the
commands the chrome issues (transport, track, filter, rate, shutdown):

- **direct**: the original file streamed with Range requests, zero server work.
  iOS takes it for plain single-audio MP4s; Android for anything ExoPlayer
  demuxes (MKV included), decided in `src/player/caps.ts`.
- **master**: the server's continuous HLS remux, anchored at `baseSec`
  (absolute position = `baseSec + player clock`, honest start read from the
  `X-Hls-Start` header). Far seeks and audio-track switches re-anchor.
- Failure ladder: direct falls back to the master once; a copy-audio master
  retries once as AAC; then the error surfaces.
- Subtitles are fetched as WebVTT and rendered as an overlay with the shared
  `parseVtt`/`activeCueText` from `@kroma/core` (same as the TV client).
- Resume + the admin playback heartbeat ride one interval
  (`src/player/heartbeat.ts`).

## Features beyond browsing

- Netflix-style Home (billboard hero, continue watching, My List, server rails),
  genre explorer (`/genres`), sort + genre filters on the Films/Series tabs.
- Quick Connect (`/connect-device`): scan the TV's QR code (the code is read
  from the authorize URL's query params) or type the 4-digit code to sign that
  device into the account, like the web client.
- Offline downloads (`src/lib/downloads/` + `/downloads`): EVERY title is
  downloadable. When the device can play the original end to end (container,
  video and every audio track, per `canRawDownload`) it is fetched RAW, byte for
  byte, at zero server cost; otherwise `GET /api/items/:id/download` remuxes it
  on the fly to a fragmented MP4 that keeps every audio track (each stream-copied
  or AAC-transcoded to match what this device decodes, sent as `?copy=`), so an
  MKV library is still downloadable on iOS with its languages intact. One
  transfer at a time; a season queues.
  - That endpoint is session-gated, so the transfer attaches the bearer itself
    via `client.authHeaders()` (media-element URLs like `streamUrl` stay public
    precisely because a `<video>` cannot send a header).
  - A finished transfer is only registered once it is proven whole: HTTP status,
    media content type, exact size for a raw copy, plausibility floor for a
    remux. The remux is chunked with no `Content-Length`, so a stream that dies
    mid-file otherwise looks like a clean finish.
  - `store.ts` reconciles the download directory against the index at startup and
    deletes orphans, so a transfer killed with the app can't strand gigabytes
    that no screen can see.
- Player: storyboard scrub previews, skip-intro + up-next from server markers,
  audio track / subtitle / speed / volume-filter sheet, double-tap seek.
  RN `Modal`s inside the landscape-locked player MUST pass
  `supportedOrientations` or iOS throws `UIApplicationInvalidInterfaceOrientation`.
- Servers + profiles (`src/lib/session/`): any number of saved servers, any
  number of remembered accounts each (device credentials in SecureStore), one
  active pointer, exactly like the TV client. `stores.ts` owns the two persisted
  lists behind one write path; `boot.ts` is the cold start. Adding a server
  probes `/api/health` (https then http, so no one types a scheme) and the
  connect screen lists what answered on the LAN via `discoverServers` from
  `@kroma/core`, which collapses one server reached through several origins on
  the `instanceId` it reports.
- Profile lock (`src/lib/biometricGate.ts`): a profile with a server PIN is
  gated by the server itself; a profile without one can opt into a standalone
  Face ID / Touch ID lock on THIS device, enforced both at the profile gate and
  on cold start. Both fail closed - an unreadable keychain reads as locked.
- Icons are `@tabler/icons-react-native`, the same family as the web client,
  wrapped in `src/player/icons.tsx` / `src/components/tabIcons.tsx`. Import
  icons via DEEP paths (`.../dist/esm/icons/IconX.mjs`, typed by
  `src/types/tabler-icons.d.ts`): the package barrel loses exports under
  Metro's Hermes live-bindings transform.

Dev auto-login: put `EXPO_PUBLIC_KROMA_SERVER` and
`EXPO_PUBLIC_KROMA_DEV_LOGIN=user:pass` in `.env.local` (gitignored) to boot
straight into a session.

## Conventions

`app.json` turns on the React Compiler, and Expo runs the app in StrictMode in
development. Two consequences worth knowing before editing state code: a ref
assigned in a render body can go stale when a render is memoized away, and a
`setState` updater that also writes to disk runs twice. Persistence therefore
lives in explicit write paths (`commitEntries`, `persistAccounts`,
`persistServers`), never inside an updater, and refs are written from effects or
those same helpers.

Files stay under 300 lines (repo-wide rule), which is why `downloads`, `session`,
`engine` and `ui` are directories rather than single modules.

## Not in the release pipeline (yet)

`.github/workflows/release.yml` builds the Synology package, desktop installers
and the TV bundles; there is no Expo/EAS job, so tagging a release ships no
mobile binary. Builds are local (`bun run ios` / `bun run android`) or a manual
EAS run until that job exists. Note that `app.json`'s `version` is independent of
the release tag: `chore(release)` bumps `server/Cargo.toml`, not this app.

## Brand assets
